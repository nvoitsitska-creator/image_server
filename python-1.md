## **1. Вступ**

### **Що ми будемо робити?**

Наше завдання — створити сервіс для перегляду та зберігання зображень у вигляді веб-додатку.
Суть проста: користувач заходить на сайт, завантажує фотки чи картинки,
а натомість отримує прямі посилання на них.
Ці посилання можна використовувати де завгодно — у соцмережах, блогах або просто надіслати друзям.

### **Які технології використовуємо?**

- Бекенд пишемо на Python — він чудово підходить для таких завдань.
- Для швидкої роздачі зображень використовуємо Nginx — він просто створений для цього.
- Все запакуємо в Docker-контейнери, щоб проект легко запускався на будь-якому комп'ютері.
- Docker Compose допоможе керувати всіма компонентами як єдиним цілим.

### **Навіщо цей проект?**

Наш сервіс — це не просто сховище зображень. Це місце, де користувачі можуть:
- Завантажувати фотки в популярних форматах (.jpg, .png, .gif).
- Бути впевненими, що їхні зображення надійно збережені.
- Отримувати зручні прямі посилання на завантажені зображення.

Ну а ми, розробники, будемо стежити за всім процесом через систему логів,
щоб знати, що робити, якщо щось піде не так.

А для навчання цей проект — чудовий шанс покращити свої навички в Python-розробці, налаштуванні веб-серверів,
роботі з Docker та взаємодії різних компонентів системи один з одним.

### **Основні завдання:**
1. Розробити Python-бекенд для обробки запитів, завантаження зображень та логування дій користувачів.

5. Реалізувати інтерфейс для взаємодії з користувачем (завантаження зображень та доступ до файлів).

## **2. Функціональні вимоги**

### Основний функціонал:

**1. Головна сторінка**

- **Маршрут:** `/`
- Функціональність:
  - Зустрічає користувача вітальним повідомленням та розповідає про сервіс.
  - Містить посилання на основні маршрути:
  - Сторінка завантаження зображень (`/upload`).
  - Каталог завантажених зображень (`/images/`).

**2. Завантаження зображень**

- **Маршрут:** `/upload`
- **Метод:** `POST`
- Функціональність:
   - Приймає файл зображення від користувача через HTTP-запит.
   - Підтримувані формати: `.jpg`, `.png`, `.gif`.
   - Обмеження розміру файлу: до 5 МБ.
   - Генерує унікальне ім'я для кожного завантажуваного зображення.
   - Зберігає зображення в папку `/images` на сервері.
   - Повертає користувачу унікальний ідентифікатор файлу та посилання на зображення.

**3. Доступ до завантажених зображень**

- **Маршрут:** `/images/<ім'я_файлу>`
- Функціональність:
   - Всі завантажені зображення мають бути доступні для перегляду та завантаження за прямими посиланнями.
   - Nginx обслуговує цей маршрут, роздаючи файли з папки `/images`.

**4. Логування дій**

- Функціональність:
  - Записує кожну дію користувача в лог-файл `app.log`.
- Формат запису:
   ```txt
   [Дата/час] Дія: повідомлення
   ```
Приклади записів:
   ```text
   [2025-01-24 14:00:00] Успіх: зображення img1.jpg завантажено.
   [2025-01-24 14:01:00] Помилка: непідтримуваний формат файлу (user_file.txt).
   ```
- Логи потрібно зберігати в окремому томі Docker.

------

### **Користувацькі сценарії**

**1. Завантаження зображення**
1. Користувач переходить на маршрут `/upload`.
2. Завантажує файл через HTTP-запит (наприклад, за допомогою форми або програми `curl`).
3. Бекенд перевіряє формат та розмір файлу.
4. Якщо файл успішно завантажено:
   - Він зберігається в папці `/images`.
   - Користувачу повертається повідомлення з унікальним посиланням на файл (наприклад, `/images/img1.jpg`).
5. Якщо файл не відповідає вимогам:
   - Користувачу повертається повідомлення про помилку (наприклад, "Непідтримуваний формат файлу").

**2. Перегляд завантажених зображень**
1. Користувач переходить за посиланням `/images/<ім'я_файлу>`.
2. Nginx повертає запитане зображення для перегляду або завантаження.

## **3. Нефункціональні вимоги**

### **1. Продуктивність**

- **Багатозадачність:** сервер легко справляється з 10 користувачами, які завантажують картинки одночасно. Жодних гальмувань та зависань!
- **Швидкість завантаження:** час обробки завантаження одного зображення (включаючи збереження та генерацію імені) не повинен перевищувати 1 секунди.
- **Миттєвий перегляд:** Nginx роздає картинки зі швидкістю світла (ну гаразд, менше 100 мілісекунд).
  Відкрили посилання — відразу побачили картинку, жодного "завантажується...". Коротше кажучи, все має працювати швидко та без затримок! ⚡

### **2. Обмеження**

- **Розмір файлів:** максимальний розмір завантажуваного зображення — 5 МБ.
- **Підтримувані формати:** тільки `.jpg`, `.png` та `.gif`. Спроба завантажити непідтримуваний формат має повертати повідомлення про помилку.
- **Обсяг зберігання:** сховище зображень не повинно перевищувати доступний простір тому Docker (за замовчуванням обмежень на обсяг не вводиться, але передбачається розумне використання дискового простору).

### **3. Надійність**

- Система має коректно обробляти некоректні запити:
- Завантажені файли, що не відповідають формату або розміру, мають викликати повернення помилки (400 Bad Request).
- У разі збою сервер має автоматично перезапускатися (мається на увазі використання Docker для автоматичного відновлення).

### **4. Масштабованість**

- Проект орієнтований на навчальні цілі, але має бути спроектований таким чином, щоб за необхідності:

## **4. Архітектура та структура**

### **1. Загальна архітектура**

Система складається з двох основних компонентів:
1. **Python-бекенд:**
- Відповідає за обробку HTTP-запитів, завантаження зображень, валідацію даних та логування.
- Виконує бізнес-логіку додатку, пов'язану з керуванням завантаженими файлами.
- Працює всередині Docker-контейнера, слухаючи запити на порту `8000`.
2. **Nginx-сервер:**
- Роздає статичні файли та завантажені зображення за маршрутом `/images/`.
- Проксує запити на Python-бекенд для інших маршрутів.
- Працює в окремому Docker-контейнері, слухаючи запити на порту `80`.

Компоненти взаємодіють через локальну мережу, створену за допомогою Docker Compose.

### **2. Взаємодія компонентів**

**Коли користувач завантажує картинку:**
1. Користувач надсилає файл через `POST /upload` на Python-бекенд.
2. Бекенд бере всю роботу на себе:
    - Перевіряє, чи все гаразд з файлом
    - Придумує йому унікальне ім'я
    - Зберігає в папку `/images` через Docker volume
    - Записує в лог, що і коли завантажили
    - Віддає користувачу посилання на його картинку

**Коли користувач хоче подивитися картинку:**
1. Запитує картинку через `GET /images/<ім'я_файлу>` у Nginx
2. Nginx шукає файл у папці `/images`
3. Якщо знайшов - віддає картинку користувачу

**Хто за що відповідає:**
- Python-бекенд — "розумна" частина: перевіряє файли, обробляє завантаження
- Nginx — "швидка" частина: роздає картинки і працює як проксі

### **3. Структура проекту**

**Директорії та файли:**
```text
project/
├── app.py                # Основний Python-бекенд
├── requirements.txt      # Залежності Python
├── Dockerfile            # Dockerfile для Python-бекенду
├── docker-compose.yml    # Конфігурація Docker Compose
├── nginx.conf           # Конфігурація Nginx
├── images/              # Папка для зберігання завантажених зображень (volume)
├── logs/                # Папка для логів (volume)
└── static/              # Додаткові статичні файли (CSS/JS, якщо застосовно)
```

**Опис ключових компонентів:**
1. **app.py:**
- Реалізує обробку HTTP-запитів:
    - `GET /` — головна сторінка.
    - `POST /upload` — завантаження зображення.
- Виконує:
    - Валідацію завантажуваного файлу (формат, розмір).
    - Збереження файлу в папку `/images`.
    - Логування дій у файл `app.log`.
  
2. **nginx.conf:**
- Роздає вміст папки `/images` за маршрутом `/images/<ім'я_файлу>`.
- Проксує запити, не пов'язані зі статикою, на Python-бекенд.

3. **compose.yaml:**
- Визначає два сервіси:
    - `app` — Python-бекенд.
    - `nginx` — Nginx-сервер.
- Створює два томи:
    - `images` — для зберігання завантажених зображень.
    - `logs` — для зберігання логів.
  
4. **images/:**
- Каталог, у який зберігаються всі завантажені файли.
- Підключається як volume для довготривалого зберігання.

5. **logs/:**
- Каталог, що містить лог-файли (`app.log`).
- Підключається як volume для збереження даних між перезапусками.

    - Зовнішній порт: `8080` (для користувачів).
    - Внутрішня мережа Docker зв'язує два контейнери:
    - Nginx звертається до Python-бекенду за адресою `http://app:8000`.

### **5. Логічна схема**

1. **Користувацький запит:**
   Користувач взаємодіє з системою через маршрути:
   - `/` — головна сторінка (обробляється бекендом).
   - `/upload` — завантаження зображення (обробляється бекендом).
   - `/images/<ім'я_файлу>` — доступ до зображення (обробляється Nginx).

2. **Обробка запитів:**
   - Nginx:
       - Роздає статичні файли та завантажені зображення.
       - Переадресовує запити на Python-бекенд, якщо маршрут не пов'язаний з інтерфейсом та зображеннями.
   - Python-бекенд:
       - Приймає і обробляє запити `/` та `/upload`.
       - Зберігає зображення та записує логи.
     
3. **Зберігання даних:**
   - Зображення:
       - зберігаються в volume, підключеному як `/images`.
   - Логи:
       - зберігаються в volume, підключеному як `/logs`.

### **6. Потік даних**

1. **Завантаження зображення:**
   - Користувач надсилає файл на маршрут `/upload`.
   - Python-бекенд:
       - Перевіряє файл.
       - Зберігає файл в `/images`.
       - Логує дію.
   - Користувачу повертається посилання виду `/images/<ім'я_файлу>`.
   
2. **Доступ до зображення:**
   - Користувач надсилає запит на `/images/<ім'я_файлу>`.
   - Nginx перевіряє файл у папці `/images` і повертає його.

      - `images` — для зберігання завантажених зображень.
      - `logs` — для зберігання логів.
  - Для керування контейнерами використовується Docker Compose.
- **Docker Compose:**
   - Конфігурація має містити:
       - Сервіс для Python-бекенду:
           - Назва: `app`.
           - Порт: `8000`.
           - Підключення томів для зображень та логів.
       - Сервіс для Nginx:
           - Назва: `nginx`.
           - Порт: `8080`.
           - Підключення тому для зображень.
           - Проксування запитів на Python-бекенд.

------

### **4. Сховище даних**

- **Зображення:**
  - зберігаються в локальній папці `/images`, підключеній через volume.
  - кожне зображення має мати унікальне ім'я, що генерується Python-бекендом.
  - обмеження:
      - Підтримувані формати: `.jpg`, `.png`, `.gif`.
      - Максимальний розмір файлу: 5 МБ.
- **Логи:**
  - логи пишуться в файл `app.log`, розташований у папці `/logs`.
  - лог-файл має бути доступний для читання між перезапусками контейнера.

### **5. Мережеві налаштування**

- Взаємодія контейнерів:
- Python-бекенд (`app`) та Nginx (`nginx`) мають бути зв'язані через внутрішню мережу Docker.
- Nginx проксує запити до Python-бекенду за адресою `http://app:8000`.
- Порти:
- Nginx: доступний на порту `8080` для користувачів.
- Python-бекенд: слухає запити на порту `8000` (всередині контейнера).
