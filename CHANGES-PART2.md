# Зміни — Частина 2: Інтеграція з PostgreSQL, виправлення помилок та рефакторинг

## Новий функціонал

### 1. Підключення PostgreSQL

**Файли:** `compose.yaml`, `app.py`, `requirements.txt`

**Опис:** Додано сервіс `db` (PostgreSQL 17) у Docker Compose. Python-бекенд підключається до бази через `psycopg2` та автоматично створює таблицю `images` при запуску.

- Сервіс `db` використовує `postgres:17` з healthcheck (`pg_isready`)
- Додано named volume `db_data` для збереження даних між перезапусками
- `app` залежить від `db` з умовою `service_healthy`
- З'єднання з БД кешується та автоматично перепідключається (`get_db_connection()`)
- Ініціалізація таблиці має 5 спроб з інтервалом 2 секунди (`init_db()`)
- При зупинці сервера з'єднання з БД закривається (`close_db()`)

```sql
CREATE TABLE IF NOT EXISTS images (
    id SERIAL PRIMARY KEY,
    filename VARCHAR(255) NOT NULL,
    original_name VARCHAR(255) NOT NULL,
    size INTEGER NOT NULL,
    file_type VARCHAR(10) NOT NULL,
    upload_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 2. Збереження метаданих при завантаженні

**Файл:** `app.py` — метод `_handle_upload()`

**Опис:** Після збереження файлу на диск, метадані записуються в БД. Якщо запис у БД не вдається — файл видаляється з диска і повертається помилка 500.

Поля, що зберігаються:
- `filename` — згенерована UUID-назва
- `original_name` — оригінальна назва від користувача
- `size` — розмір файлу в байтах
- `file_type` — розширення без крапки (jpg, png, gif)

---

### 3. Сторінка списку зображень (`/images-list`)

**Файл:** `app.py` — метод `_handle_images_list()`

**Опис:** Новий GET-маршрут, який рендерить HTML-сторінку з таблицею зображень з БД.

- Таблиця з колонками: Назва файлу, Оригінальна назва, Розмір (КБ), Дата завантаження, Тип файлу, Дія
- Назва файлу є посиланням на `/images/<filename>`
- Кнопка "Видалити" для кожного запису
- Порожній стан: "Немає завантажених зображень"
- Всі дані користувача екрануються через `html.escape()`
- Інлайн CSS, стилізований під загальний дизайн проєкту

---

### 4. Пагінація

**Файл:** `app.py` — метод `_handle_images_list()`

**Опис:** Список зображень розбитий на сторінки по 10 записів (`ITEMS_PER_PAGE = 10`).

- Параметр `?page=N` в URL (за замовчуванням 1)
- SQL: `LIMIT 10 OFFSET (page-1)*10`, сортування за `upload_time DESC`
- Кнопки "Попередня сторінка" / "Наступна сторінка"
- Кнопки деактивуються на межах (перша/остання сторінка)
- Відображається номер поточної сторінки та загальна кількість

---

### 5. Видалення зображень за ID (`/delete/<id>`)

**Файл:** `app.py` — метод `_handle_delete_by_id()`

**Опис:** Новий POST-маршрут для видалення зображення через веб-інтерфейс.

- Витягує `id` з URL, валідує як ціле число
- Знаходить `filename` в БД за ID, повертає 404 якщо не знайдено
- Видаляє запис з БД та файл з диска
- Якщо файл відсутній на диску — логує попередження, але не повертає помилку
- Після видалення перенаправляє (303) на `/images-list`

---

### 6. Видалення зображень через DELETE-запит (`DELETE /images/<filename>`)

**Файли:** `app.py` — методи `do_DELETE()`, `_handle_delete()`; `static/image-uploader/js/images.js`

**Опис:** Додано новий HTTP-метод `do_DELETE` та обробник `_handle_delete`:

- Приймає DELETE-запити на `/images/<filename>`
- Перевіряє, що шлях не виходить за межі `IMAGES_DIR` (захист від path traversal через `os.path.realpath()`)
- Видаляє файл з диска та запис з БД
- Повертає JSON-відповідь з результатом

**Фронтенд (`images.js`):**
- Додано функцію `deleteFromServer(imageUrl)` — надсилає DELETE-запит на сервер перед видаленням з `localStorage`
- URL зображення зберігається в `data-url` атрибуті кнопки видалення

### 7. Резервне копіювання

**Файл:** `backup.sh` (новий)

**Опис:** Shell-скрипт для створення бекапу бази даних через `pg_dump`.

```bash
./backup.sh
# Створює файл: backups/backup_20260319_143000.sql
```

- Створює папку `backups/` якщо не існує
- Ім'я файлу включає дату та час
- Для відновлення: `docker exec -i image-server-db psql -U postgres images_db < backups/backup_file.sql`

## Завдання для виправлення

### 10. Сторінка `/images-list` генерує HTML на бекенді

**Файл:** `app.py` — метод `_handle_images_list()` (рядки 236-435)

**Проблема:** Метод `_handle_images_list()` генерує повну HTML-сторінку (стилі, розмітку, пагінацію) прямо в Python-коді через конкатенацію рядків. Це суперечить архітектурі решти проєкту, де використовуються статичні HTML-файли (`upload.html`, `images.html`) + JavaScript для роботи з API.

**Завдання:**
- Створити статичний HTML-файл для сторінки списку зображень (аналогічно до `upload.html` / `images.html`)
- Додати JSON API endpoint для отримання списку зображень з БД (з пагінацією)
- Реалізувати рендеринг таблиці та пагінації на клієнті через JavaScript

---

### 11. Редирект з головної сторінки

**Файл:** `app.py` — метод `do_GET()` (рядки 119-121)

**Проблема:** Маршрут `/` робить редирект (302) на `/upload`. Через це файл `static/index.html` (головна/лендінг сторінка) ніколи не відображається користувачу — немає жодного маршруту, який би її віддавав.

**Завдання:**
- Виправити маршрутизацію так, щоб головна сторінка (`index.html`) була доступна за маршрутом `/`

---

### 12. Deprecated модуль `cgi`

**Файл:** `app.py` — рядок 6 (`import cgi`), рядки 458-466 (`cgi.FieldStorage`)

**Проблема:** Модуль `cgi` deprecated з Python 3.11 і **повністю видалений** у Python 3.13+. Проєкт використовує `cgi.FieldStorage` для парсингу multipart/form-data при завантаженні файлів. При оновленні Python до версії 3.13+ сервер перестане працювати.

**Завдання:**
- Замінити `cgi.FieldStorage` на сучасну альтернативу (наприклад, пакет `python-multipart`)
- Оновити `requirements.txt` відповідно
- Переконатися, що завантаження файлів працює як раніше

---

### 13. F5 перехоплює оновлення сторінки

**Файл:** `static/image-uploader/js/common.js` — функція `registerKeyboardShortcuts()` (рядки 29-36)

**Проблема:** Функція перехоплює натискання клавіші F5 (`event.preventDefault()`) і замість стандартного оновлення сторінки робить редирект на інший URL. Це ламає очікувану поведінку браузера — користувач не може оновити сторінку через F5.

**Завдання:**
- Прибрати перехоплення F5 з `registerKeyboardShortcuts()`
- Залишити тільки Escape як клавіатурне скорочення (або замінити F5 на іншу комбінацію)

---

### 14. Race condition при видаленні зображення

**Файл:** `static/image-uploader/js/images.js` — функція `addDeleteListeners()` (рядки 108-123)

**Проблема:** При натисканні кнопки "Видалити" функція `deleteFromServer()` викликається, але її результат ігнорується. Навіть якщо сервер повернув помилку (наприклад, файл вже не існує або мережа недоступна), зображення все одно видаляється з `localStorage`. Це створює неузгодженість: користувач бачить що зображення видалено, але файл залишається на сервері.

```js
// Поточний код (результат deleteFromServer ігнорується):
await deleteFromServer(imageUrl);
storedFiles.splice(indexToDelete, 1);  // виконується завжди
```

**Завдання:**
- Перевіряти результат `deleteFromServer()` перед видаленням з `localStorage`
- Показувати повідомлення про помилку, якщо видалення на сервері не вдалося

## Перевірка

1. `docker compose up --build` — всі 3 сервіси запускаються, таблиця створюється
2. Відкрити `/` — перенаправляє на `/upload`
3. Завантажити зображення через `/upload` — перевірити запис у БД:
   ```bash
   docker exec -it image-server-db psql -U postgres images_db -c "SELECT * FROM images"
   ```
4. Перевірити копіювання URL кнопкою "COPY" (працює через HTTP)
5. Відкрити `/images-list` — таблиця показує завантажене зображення
6. Натиснути "Видалити" на `/images-list` — зображення видалено з БД та диска
7. Видалити зображення через кнопку на сторінці Images — файл видаляється з сервера
8. Завантажити 11+ зображень — перевірити пагінацію
9. Натиснути "DB Gallery" на всіх сторінках — веде на `/images-list`
10. `./backup.sh` — перевірити створення `.sql` файлу в `backups/`
